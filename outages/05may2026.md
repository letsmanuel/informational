Postmortem: Full VPS CPU Saturation Incident (Hetzner Ubuntu 4GB)

Summary

On May 05, the VPS experienced a complete system-wide degradation event caused by a single Python process entering a high-frequency exception loop. The system reached sustained 100% CPU utilization across both virtual CPUs, leading to SSH unresponsiveness, Docker instability, container runtime failures, and monitoring gaps across external systems (including Hetzner dashboard reporting “system unavailable”).

Root cause was traced to a Discord bot process (anticheat-bot) repeatedly failing SQLite write operations under concurrent access conditions.

⸻

Impact

* SSH became unresponsive
* Docker daemon (dockerd) and container runtime (containerd) degraded
* Container health checks failed or timed out
* System monitoring (Beszel + Hetzner panel) showed gaps and “system unavailable”
* All hosted services became unreachable or unstable

⸻

Affected System

* Host: ubuntu-4gb-nbg1-4
* Primary process: python[3640301]
* Application: /opt/anticheat-bot/
* Runtime: Python 3.12 + aiosqlite + discord.py

⸻

Root Cause Analysis

Primary cause

A single Python process:

python[3640301]
/opt/anticheat-bot/

entered a tight exception loop caused by SQLite write contention.

Core error:

sqlite3.OperationalError: database is locked

This occurred during:

discord.client: Ignoring exception in on_message
await db.commit()

⸻

Failure mechanism

1. Discord event (on_message) triggered at high frequency
2. Bot attempted SQLite write operation via aiosqlite
3. SQLite returned database is locked
4. Exception was thrown repeatedly without delay or backoff
5. Async event loop continuously retried the operation
6. CPU utilization spiked due to tight retry loop
7. No scheduling time remained for system processes

⸻

System-wide impact explanation

Due to sustained CPU saturation:

* Kernel scheduler could not allocate CPU time to system services
* SSH daemon became unresponsive
* Docker daemon socket requests timed out
* container runtime (containerd) lost internal stream synchronization
* monitoring agents failed to execute on schedule

This resulted in a full observability and access failure across the system.

⸻

Contributing factors

* SQLite used in concurrent async write environment
* Lack of write locking or serialization
* No retry backoff on database lock errors
* High-frequency Discord event handling (on_message)
* No CPU limits applied to the process
* Single process allowed unrestricted CPU usage

⸻

Resolution

* VPS required forced restart to recover
* No reliable process-level CPU history available post-event due to system starvation
* Service state was not recoverable in-place

⸻
Conclusion

The incident was caused by a single Python process (anticheat-bot) entering a high-frequency exception loop due to SQLite write contention. The resulting CPU spin condition fully saturated the VPS, leading to cascading failure across system services, container runtime, and monitoring infrastructure.

No evidence of kernel failure or external compromise was found.
