Incident Postmortem: VPS CPU Saturation

05 May 2026 | Host: ubuntu-4gb-nbg1-4 | Severity: SEV-1

Overview

A single process caused sustained full CPU saturation on a small VPS instance. The system did not crash, but became unresponsive due to complete scheduler starvation across all services.

The issue originated from a Python Discord bot using SQLite for persistence. Under concurrent write load, the database entered a locked state. The application did not implement backoff or throttling, resulting in a continuous retry loop inside the event handler.

At peak load, both virtual CPUs were fully saturated (200% combined utilization). During this period, external monitoring tools triggered alerts and staff were paged due to multiple service failures and lost health checks.

Timeline

System was operating normally with stable load and no visible resource pressure.

Alerting and escalation

At peak CPU usage (200%), external monitoring systems detected service outages and triggered alerts. On-call staff were paged due to simultaneous failures across multiple services and missing health checks.

Root cause

The failing process was: 

python[3640301]
/opt/anticheat-bot/

The application repeatedly processed Discord events and attempted SQLite writes. When the database locked, the process retried immediately without delay, creating a continuous CPU-bound loop.

This resulted in a sustained spin condition that consumed all available CPU resources.

Impact

  • Full CPU saturation (200% across 2 vCPUs)
  • SSH became unresponsive
  • Docker daemon and container runtime degraded
  • Monitoring systems failed or showed gaps
  • External alerting systems paged staff during peak outage

Conclusion

A single unbounded retry loop in a user-space application was sufficient to fully saturate CPU resources on the host. The system remained running but effectively unusable due to scheduler starvation.
